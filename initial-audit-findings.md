# Initial Audit Findings

`npm run typecheck` passes. I did not find any executable test suite in the repo.

## 1. Critical Issues

1. Anonymous RLS is still effectively open on core tables. Unauthenticated clients can read all jobs and fully insert/update/delete `sessions`, `notes`, `materials`, `attachments`, and `job_activity_events` through the public anon key: [backend/supabase/migrations/20250411120000_create_jobs.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20250411120000_create_jobs.sql:15), [backend/supabase/migrations/20260412200412_init_field_book_phase1.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260412200412_init_field_book_phase1.sql:168), [backend/supabase/migrations/20260413120000_phase1_job_economic_session_worklog.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260413120000_phase1_job_economic_session_worklog.sql:129), [backend/supabase/migrations/20260413120001_rename_materials_and_job_activity_events.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260413120001_rename_materials_and_job_activity_events.sql:59). The README explicitly says these same migrations are the source of truth for hosted projects, so this is not safely confined to local dev: [backend/supabase/README.md](/Users/johnveltri/dev/fieldbook/backend/supabase/README.md:3).

2. Child-row ownership is not tied to parent ownership, so data integrity across `jobs`, `sessions`, `notes`, `materials`, `attachments`, and `job_activity_events` is breakable. The authenticated policies only check each child row’s `user_id`, not whether its `job_id` or `session_id` belongs to that same user: [backend/supabase/migrations/20260413120003_authenticated_own_policies_sessions_notes_materials_attachments_job_activity.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260413120003_authenticated_own_policies_sessions_notes_materials_attachments_job_activity.sql:18), [backend/supabase/migrations/20260412200412_init_field_book_phase1.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260412200412_init_field_book_phase1.sql:67), [backend/supabase/migrations/20260413120000_phase1_job_economic_session_worklog.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260413120000_phase1_job_economic_session_worklog.sql:55), [backend/supabase/migrations/20260413120001_rename_materials_and_job_activity_events.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260413120001_rename_materials_and_job_activity_events.sql:40). As written, a user can attach their own child rows to any known job/session UUID, which will become a serious corruption problem once IDs are exposed beyond a single-screen prototype.

## 2. Medium-Risk Issues

1. The seeded demo job is globally readable by every authenticated user and globally updateable by every authenticated user before claim, which means one shared row can become everyone’s default landing job. The policies allow shared access: [backend/supabase/migrations/20260413120004_authenticated_select_seed_demo_job.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260413120004_authenticated_select_seed_demo_job.sql:6), [backend/supabase/migrations/20260417120001_jobs_update_seed_demo.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260417120001_jobs_update_seed_demo.sql:3); the app then picks the most recently updated visible job with no special handling: [packages/api-client/src/jobs.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/jobs.ts:31). In any shared environment, one person editing the demo row can change what other users see first.

2. Job status mapping is inconsistent with the schema comments. The DB says “paid” is derived from payment state plus completed work, but the client returns `paid` whenever `job_payment_state === 'paid'`, even if the job is still `in_progress` or `not_started`: [backend/supabase/migrations/20260412200412_init_field_book_phase1.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260412200412_init_field_book_phase1.sql:51), [packages/api-client/src/jobDetail.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/jobDetail.ts:100). That will produce incorrect pills and filtering as soon as partial-payment flows exist.

3. `fetchJobDetail` silently ignores failures from notes/materials/activity queries and will render partial data as if it were valid. The function checks `jobErr` and `sErr`, but not the errors from `notesQ`, `materials`, or `job_activity_events`: [packages/api-client/src/jobDetail.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/jobDetail.ts:159), [packages/api-client/src/jobDetail.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/jobDetail.ts:192). If an RLS rule, migration drift, or query regression breaks one subquery, the screen will quietly show empty buckets instead of surfacing a real failure.

4. Notes and materials attached to discarded sessions can disappear from the UI without being deleted. The query includes job-level rows broadly, but the bucket rendering only emits session buckets for `activeSessions`, so rows anchored to discarded sessions are neither “unassigned” nor rendered under a session: [packages/api-client/src/jobDetail.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/jobDetail.ts:180), [packages/api-client/src/jobDetail.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/jobDetail.ts:246), [packages/api-client/src/jobDetail.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/jobDetail.ts:276). That is silent data loss from the user’s perspective.

5. Soft delete is only half-modeled. `jobs.deleted_at` exists and the main indexes already assume only non-deleted rows matter, but the API never filters it out when selecting or updating jobs: [backend/supabase/migrations/20260412200412_init_field_book_phase1.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260412200412_init_field_book_phase1.sql:48), [backend/supabase/migrations/20260412200412_init_field_book_phase1.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260412200412_init_field_book_phase1.sql:59), [packages/api-client/src/jobs.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/jobs.ts:34), [packages/api-client/src/jobs.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/jobs.ts:84). Once soft delete is used, deleted jobs can still be selected as the default job and edited.

6. Validation on job editing is too weak for a financial record. The UI accepts arbitrary revenue strings, parses them permissively, and turns malformed input into `null`, which can wipe out revenue; only `short_description` is protected at the DB layer, while `jobType`, `serviceAddress`, and `customerName` can be saved as meaningless whitespace: [apps/mobile-expo/src/screens/JobDetailScreen.tsx](/Users/johnveltri/dev/fieldbook/apps/mobile-expo/src/screens/JobDetailScreen.tsx:198), [apps/mobile-expo/src/screens/JobDetailScreen.tsx](/Users/johnveltri/dev/fieldbook/apps/mobile-expo/src/screens/JobDetailScreen.tsx:225), [apps/mobile-expo/src/components/ds/EditJobBottomSheet.tsx](/Users/johnveltri/dev/fieldbook/apps/mobile-expo/src/components/ds/EditJobBottomSheet.tsx:124), [backend/supabase/migrations/20260412200412_init_field_book_phase1.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260412200412_init_field_book_phase1.sql:53). This is the kind of input path that creates bad rows early and forces cleanup migrations later.

7. Several important domain invariants are missing from the schema. `collected_cents` is only checked for non-negativity, so it can exceed revenue or coexist with `job_payment_state='paid'` while still being null; `materials` allows `quantity`, `unit_cost_cents`, and `total_cost_cents` to disagree with each other: [backend/supabase/migrations/20260413120000_phase1_job_economic_session_worklog.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260413120000_phase1_job_economic_session_worklog.sql:24), [backend/supabase/migrations/20260412200412_init_field_book_phase1.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260412200412_init_field_book_phase1.sql:137). That is manageable in a prototype, but it will become expensive once reporting or invoicing depends on these fields.

## 3. Low-Risk Improvements

1. The shared types are already too UI-shaped for the domain. `JobDetailViewModel` drops raw IDs, timestamps, and money fields for notes/materials, while the base `Job` type omits DB-backed fields like `service_address` and `job_type`: [packages/shared-types/src/jobDetailView.ts](/Users/johnveltri/dev/fieldbook/packages/shared-types/src/jobDetailView.ts:15), [packages/shared-types/src/index.ts](/Users/johnveltri/dev/fieldbook/packages/shared-types/src/index.ts:22). That will make edit/delete flows and richer syncing awkward because the client model is losing source-of-truth data.

2. The Supabase client is completely untyped against the actual schema. `FieldbookSupabaseClient` is just `SupabaseClient` with no generated `Database` generic: [packages/api-client/src/client.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/client.ts:3). As migrations evolve, column and enum drift will not be caught at compile time.

3. `fetchJobDetail` is unbounded and will not scale cleanly. It loads all sessions, all notes for the job/session set, and all materials for the job/session set in one screen request, with `select('*')` on materials and no pagination: [packages/api-client/src/jobDetail.ts](/Users/johnveltri/dev/fieldbook/packages/api-client/src/jobDetail.ts:172). That is fine for demo data, but it will become the first performance bottleneck as jobs accumulate history.

4. Documentation already disagrees with behavior. The README says seeded rows do not auto-attach and that users only see their own jobs, but the seed-demo policies deliberately expose a shared demo job to every authenticated user: [backend/supabase/README.md](/Users/johnveltri/dev/fieldbook/backend/supabase/README.md:41), [backend/supabase/migrations/20260413120004_authenticated_select_seed_demo_job.sql](/Users/johnveltri/dev/fieldbook/backend/supabase/migrations/20260413120004_authenticated_select_seed_demo_job.sql:6). That kind of drift usually precedes operational mistakes.

## 4. Recommended Tests To Add

1. SQL/RLS tests proving anonymous users cannot read or mutate production data for `jobs`, `sessions`, `notes`, `materials`, `attachments`, and `job_activity_events`.
2. SQL/RLS tests proving an authenticated user cannot create or update child rows against another user’s `job_id` or `session_id`.
3. SQL tests for demo-job behavior: visibility, update permissions, and whether it can incorrectly outrank owned jobs in the default picker.
4. Unit tests for `mapWorkStatus` covering every combination of `job_work_status` and `job_payment_state`.
5. Unit tests for `fetchJobDetail` that assert subquery errors are surfaced, not converted into empty buckets.
6. Unit tests for note/material bucketing with unassigned rows, rows with both `job_id` and `session_id`, and rows attached to discarded sessions.
7. UI/API tests for edit-job validation: blank title rejected, whitespace normalized, malformed revenue does not overwrite a valid existing value, and negative revenue is blocked.
8. Integration tests for soft delete semantics: `deleted_at` jobs should not be chosen by `fetchFirstJobIdForCurrentUser`, loaded by detail screens, or updated by normal flows.
